# Архитектура проекта «Second Brain» (Clarity)

Документ простыми словами объясняет, как устроен бот, из каких частей состоит,
как данные «текут» внутри, и **почему** приняты именно такие технические решения.

---

## 1. Что это за проект

**Second Brain** — это Telegram-бот, который работает как «второй мозг»:
человек скидывает ему мысли (текстом, голосом или фото), а бот:

- сохраняет их как заметки и сам придумывает краткое описание и теги;
- умеет искать по заметкам «по смыслу» и отвечать на вопросы;
- находит в заметках даты и события и **напоминает** о них заранее;
- раз в неделю присылает **сводку** о том, что было важного;
- работает в **часовом поясе** конкретного пользователя;
- распознаёт **изображения** (текст на картинке + описание).

Главная идея: пользователю не нужно ничего структурировать вручную —
достаточно «выгрузить мысль», а всё остальное берёт на себя бот и ИИ.

---

## 2. Стек технологий и почему он выбран

| Технология | Зачем | Почему именно она |
|-----------|-------|-------------------|
| **Python 3.12** | язык проекта | простой, много готовых библиотек для ИИ и Telegram |
| **aiogram 3.x** | работа с Telegram | современная **асинхронная** библиотека: бот может обрабатывать много сообщений одновременно, не «зависая» на медленных запросах к ИИ |
| **PostgreSQL** | хранение заметок, настроек, напоминаний | надёжная база, умеет фильтры по дате/тегам, переживает перезапуски |
| **Qdrant** | «умный» поиск по смыслу | специализированная векторная база — ищет похожие по смыслу заметки, а не по точным словам |
| **OpenAI** | эмбеддинги, распознавание голоса (Whisper), анализ картинок | у OpenAI лучшие модели для этих задач |
| **Claude (через routerai.ru)** | обработка текста | сильная модель для русского текста; routerai даёт к ней доступ из России |
| **Docker Compose** | запуск всего вместе | одной командой поднимаются бот + 2 базы данных, не нужно настраивать вручную |

> **Ключевое слово — «асинхронность».** Запрос к ИИ может занимать несколько секунд.
> Если бы код был синхронным, бот «замирал» бы для всех на это время.
> Асинхронный код (`async/await`) позволяет в это время обслуживать других пользователей.

## 3. Общая структура кода

Код разбит на слои — каждый отвечает за своё. Это упрощает понимание и доработку:

```
app/
├── main.py            # Точка входа: запускает бота и фоновый планировщик
├── config/            # Настройки и «живые» (изменяемые на ходу) значения
│   ├── settings.py    # Все параметры из .env (ключи, адреса баз, модели)
│   └── runtime.py     # Ключи API, которые можно менять без перезапуска
├── models/            # Описание данных (что такое «Заметка», «Напоминание», «Настройки»)
│   ├── note.py
│   └── settings.py
├── db/                # Работа с базами данных
│   ├── postgres.py    # Заметки, настройки, напоминания (SQL)
│   └── qdrant.py      # Векторы для поиска по смыслу
├── services/          # Бизнес-логика (главная «начинка»)
│   ├── llm.py         # Все обращения к ИИ (Claude и OpenAI)
│   ├── stt.py         # Распознавание голоса
│   ├── notes.py       # Сохранение/обновление заметок + создание напоминаний
│   ├── search.py      # Поиск по смыслу и ответы ИИ
│   ├── summary.py     # Сборка недельной сводки
│   ├── scheduler.py   # Фоновый цикл: рассылка напоминаний и сводок
│   └── timezones.py   # Часовые пояса пользователей
└── bot/
    ├── handlers.py    # Все кнопки, команды и сценарии общения с пользователем
    └── middleware.py  # Проверка доступа (белый список пользователей)
```

**Почему так разделено?** Принцип «каждый файл — про одно».
Например, если завтра нужно поменять модель ИИ — правим только `services/llm.py`,
а не ищем обращения к ИИ по всему проекту. Если меняем внешний вид кнопок —
трогаем только `handlers.py`.

## 4. Почему два разных ИИ-провайдера

Это одно из главных архитектурных решений. ИИ-задачи разные, и ни один провайдер
не закрывает все идеально:

- **Claude (через routerai.ru)** — для **текста**: краткие описания, теги, ответы
  на вопросы, сводки, извлечение событий из заметок. Claude хорошо понимает
  русский язык и рассуждает, а routerai обеспечивает к нему доступ из России.
- **OpenAI** — для того, что Claude **не умеет**:
  - **эмбеддинги** (превращение текста в числовой «вектор смысла» для поиска);
  - **Whisper** (распознавание речи из голосовых);
  - **vision** (анализ изображений).

Технически это сделано через **два клиента** в `services/llm.py`:
`get_text_client()` (Claude/routerai) и `get_openai()` (OpenAI). Оба используют
один и тот же SDK от OpenAI, потому что routerai **совместим с OpenAI API** —
это сильно упростило код: не нужны разные библиотеки.

> **Запасной вариант:** если ключ routerai не задан, бот не падает, а временно
> использует для текста модель OpenAI. Это сделано, чтобы один забытый ключ
> не «ронял» весь сервис.

## 5. Как хранятся заметки: две базы вместо одной

Заметка живёт сразу в **двух** местах, и это намеренно:

1. **PostgreSQL** хранит «саму заметку»: текст, описание, теги, дату, тип
   (текст/голос/фото). Отсюда берутся данные для списков, фильтров по дате и тегам.
2. **Qdrant** хранит **вектор** заметки — числовой отпечаток её смысла (1536 чисел).

**Зачем вектор?** Обычный поиск ищет по точным словам. Векторный поиск ищет
**по смыслу**: на вопрос «что я записывал про отдых» он найдёт заметку
«хочу съездить к морю», даже если слова «отдых» там нет. Это работает так:

```
Текст заметки ──(OpenAI эмбеддинг)──► вектор ──► хранится в Qdrant
Вопрос пользователя ──(тот же эмбеддинг)──► вектор ──► Qdrant находит ближайшие
```

Postgres и Qdrant связаны общим `id` заметки. При удалении заметки она убирается
из обоих хранилищ (см. `notes.delete_note_full`).

## 6. Главный сценарий: что происходит, когда вы пишете боту

Возьмём текстовое сообщение. Путь данных:

```
Пользователь пишет текст
      │
      ▼
handlers.handle_text  ──► сохраняет текст «в ожидании» и показывает кнопки:
                          «✅ Запомнить», «🔍 Найти», «🤖 Спросить у ИИ»
      │
      ▼ (нажата «Запомнить»)
notes.process_and_save_note
      ├─ Claude делает краткое описание + теги
      ├─ OpenAI делает эмбеддинг
      ├─ заметка пишется в PostgreSQL
      ├─ вектор пишется в Qdrant
      └─ Claude ищет в тексте даты/события → создаёт напоминания
```

Голос и фото идут почти так же, только перед этим:
- **голос** → скачивается → Whisper превращает в текст;
- **фото** → скачивается → vision-модель распознаёт текст и описывает картинку.

После этого они попадают в тот же общий поток «показать кнопки → сохранить/искать/спросить».

> **Почему через кнопки, а не сразу сохранять?** Одно и то же сообщение может быть
> и заметкой, и вопросом. Бот не угадывает за пользователя, а спрашивает —
> это понятнее и снижает число ошибок.

## 7. Умные напоминания

Это «фишка» проекта: пользователю **не нужно** отдельно заводить напоминание.
Он просто пишет мысль, а бот сам находит в ней дату.

**Как это работает:**
1. При сохранении заметки `notes._create_reminders_for_note` отправляет текст в Claude
   с просьбой: «найди будущие события и верни их дату/время».
2. Важно: Claude получает **текущее локальное время и часовой пояс пользователя**,
   поэтому «в среду в 19:00» правильно превращается в конкретную дату.
3. Время события переводится в UTC и сохраняется в таблицу `reminders`. Сразу же
   считается, **когда напомнить**: `время напоминания = время события − выбранный интервал`
   (по умолчанию за 24 часа, можно поменять в настройках на 12/6/3/1 час или 30 минут).

**Как напоминание «выстреливает»:** см. раздел про планировщик ниже.

Список ближайших напоминаний и их удаление — команда `/reminders`.

> **Почему дату вычисляет ИИ, а не код с регулярными выражениями?**
> Люди пишут даты сотней способов: «завтра», «в следующий вторник», «через 3 дня»,
> «25-го в обед». Перебрать все варианты кодом почти невозможно, а ИИ понимает
> естественный язык «из коробки».

## 8. Планировщик — «будильник» бота

Напоминания и недельные сводки должны приходить **сами**, без действий пользователя.
За это отвечает `services/scheduler.py` — это бесконечный фоновый цикл, который
запускается при старте бота (`main.on_startup`) и каждую минуту проверяет:

1. **Напоминания:** есть ли в базе напоминания, у которых «время напомнить» уже наступило?
   Если да — отправляет сообщение и помечает напоминание как отправленное.
2. **Сводки:** для каждого пользователя смотрит его местное время. Если совпали
   выбранный день недели и час — собирает и присылает сводку за 7 дней.
   Чтобы сводка не пришла дважды за день, проверяется дата последней отправки.

**Почему простой `asyncio`-цикл, а не отдельный планировщик (Celery, APScheduler)?**
- Бот — небольшой, пользователей немного. Тяжёлые инструменты усложнили бы запуск
  и потребовали бы ещё один сервис (брокер очередей).
- `asyncio.sleep(60)` внутри того же процесса — это надёжно, бесплатно и понятно.
- Каждая итерация обёрнута в защиту от ошибок: если одна рассылка упала, цикл
  не останавливается и продолжает работать.

Минус подхода: если бот выключен в момент напоминания, оно уйдёт при следующем
запуске (когда «время напомнить» уже прошло). Для личного бота это приемлемо.

## 9. Часовые пояса

Без учёта пояса напоминание «завтра в 9 утра» пришло бы в чужое время. Поэтому:

- При первом `/start` бот спрашивает часовой пояс (кнопки популярных зон или
  ручной ввод смещения вроде `+3`).
- Пояс хранится в `user_settings` как название по стандарту IANA (`Europe/Moscow`).
- Все даты — и показ заметок, и расчёт напоминаний, и время сводки — считаются
  в поясе пользователя через стандартный модуль Python `zoneinfo`.

**Почему храним название пояса, а не просто «+3»?** Название учитывает переходы
на летнее/зимнее время и исторические изменения автоматически. Для ручного ввода
смещение превращается в эквивалентную зону (`Etc/GMT-3`).

Чтобы не дёргать базу на каждое форматирование даты, пояс кешируется в памяти
(`timezones.get_tz` с кешем), а при смене пояса кеш сбрасывается.

## 10. Изображения и подписи

Когда приходит фото:
1. Бот скачивает картинку и отправляет в vision-модель OpenAI.
2. Модель **сначала дословно** выводит текст с картинки (без правок и оценок),
   затем краткое описание — так удобнее, если на фото чек, объявление, конспект.
3. Если у фото есть **подпись-команда** (Telegram это позволяет), бот понимает
   намерение (заметка / напоминание / вопрос ИИ / поиск) и выполняет его сразу.
   Если намерение неясно — уточняет кнопками.

Подпись и распознанный текст **объединяются**: так, если в подписи есть дата
(«напомни про это в пятницу»), напоминание создастся корректно.

## 11. Настройки и доступ

- **`/settings`** — пользовательские настройки: часовой пояс, за сколько напоминать,
  включение и расписание недельной сводки. Для администратора там же показываются
  кнопки смены ключей ИИ.
- **`/admin`** — управление белым списком пользователей и ключами API (только админ).
- **Белый список** (`bot/middleware.py`): бот отвечает только разрешённым
  пользователям. Если список пуст — отвечает всем.

**Ключи API можно менять «на лету»** (`config/runtime.py`): админ присылает новый
ключ, и бот пересоздаёт клиентов ИИ без перезапуска. Это удобно, если ключ
протух или закончились деньги. Минус: изменение живёт в памяти и сбрасывается
при перезапуске к значению из `.env` — это осознанный компромисс между удобством
и простотой (не храним секреты в базе).

## 12. Схема базы данных (PostgreSQL)

```
notes                         user_settings                 reminders
─────────                     ─────────────                 ─────────
id (PK)                       user_id (PK)                  id (PK)
user_id                       timezone                      user_id
created_at                    summary_enabled               note_id
note_type (text/voice/image)  summary_weekday               title
full_text                     summary_hour                  event_at   (когда событие)
summary                       last_summary_at               remind_at  (когда напомнить)
tags (JSON)                   reminder_lead_minutes         sent       (отправлено?)
                                                            created_at
```

Новые таблицы и колонки создаются автоматически при старте
(`postgres.init_tables`), включая «мягкие миграции» через
`ALTER TABLE ... ADD COLUMN IF NOT EXISTS` — поэтому обновление кода не ломает
существующую базу и не требует ручных действий.

## 13. Как всё это запускается (Docker)

`docker-compose.yml` описывает три сервиса, которые поднимаются вместе:

- **bot** — сам бот (наш код);
- **postgres** — база заметок и настроек;
- **qdrant** — векторная база для поиска.

Бот стартует только после того, как Postgres готов (`depends_on` + healthcheck),
и при падении автоматически перезапускается (`restart: unless-stopped`).
Секреты и параметры берутся из файла `.env` (см. `.env.example`).

Запуск и просмотр логов:
```bash
docker compose up -d --build
docker compose logs -f bot
```

## 14. Главные принципы, которыми руководствовались

1. **Разделяй ответственность.** Каждый модуль — про одну задачу. Легче читать,
   тестировать и менять.
2. **Используй правильный инструмент для задачи.** Postgres — для структуры,
   Qdrant — для смысла, Claude — для текста, OpenAI — для голоса/картинок/векторов.
3. **Не угадывай за пользователя.** Где намерение неоднозначно (сохранить или
   спросить?) — бот спрашивает кнопками.
4. **Переиспользуй, не дублируй.** Общие действия (сохранить/найти/спросить)
   вынесены в helper-функции и используются и кнопками, и обработчиком фото.
5. **Будь устойчивым к сбоям.** Ошибка одного напоминания не роняет планировщик;
   отсутствие ключа routerai не роняет бота; лишние переменные в `.env` игнорируются.
6. **Простота важнее преждевременной оптимизации.** Пока пользователей немного,
   фоновый `asyncio`-цикл лучше тяжёлой инфраструктуры очередей.

## 15. Что можно улучшить в будущем

- **Постоянное хранение ключей** (сейчас смена ключа живёт только до перезапуска).
- **Пропущенные напоминания**: досылать те, что выпали на время простоя бота,
  с пометкой «просрочено».
- **Несколько языков интерфейса** (сейчас всё на русском, строки зашиты в коде).
- **Тесты** на ключевые сценарии (сохранение, поиск, расчёт времени напоминаний).
- **Ограничение нагрузки** на ИИ (rate limiting), чтобы не упереться в лимиты API.
